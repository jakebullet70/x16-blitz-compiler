**GPC = Greased Piglet Compiler**: a self-hosted X16 BASIC → P-code compiler that runs on the Commander X16 (65C02, KERNAL R49), modeled on the 1986 SuperSoft Blitz!. Home dir: `c:\dev\CmdrX16\dos_tools\X16-GPCompiler`.

Locked decisions (from the user):
- Input = classic X16 BASIC V2 (tokenized on disk). Output = compact P-code run by a small bundled runtime VM, emitting a standalone `.PRG`.
- Implementation = hybrid: Prog8 front-end + 65C02 asm for hot paths. Cross-built on PC (prog8c), artifact runs on X16.
- Approach = **fresh project that ports the proven foundation from the sibling attempt `../BLITZ-COMPILER`** (see [[blitz-x16-prior-attempt]]) and redesigns the two areas that attempt gave up on: ROM string GC and command pass-through (both now proven — see [[gpc-gating-requirement]]).

The full plan is at `C:\Users\Admin\.claude\plans\going-to-create-a-concurrent-koala.md`. Toolchain + ROM-call details in [[x16-toolchain]] and [[x16-rom-internal-calls]].

Progress: **Phases 0, 1, 2, 3 DONE** (2026-07-07); Phase 4 largely covered. Phase 0 = skeleton/toolchain; Phase 1 = ported the foundation verbatim, renamed blitzc→gpc. Phase 2 = BASIC-format string storage collected by ROM `garba2` (custom GC removed): new module `src/runtime/bstr.p8`, `vm.sstack` holds descriptor addresses, string arrays are real BASIC arrays, memory map at `image_top = max(sys.progend(), datatop)`. Phase 3 = command pass-through real in the compiler: `OP_PASSTHRU`=64 hands a tokenized statement to ROM `gone3` (bank 4) via a low-RAM `passbuf`; compiler routes keyword statements it doesn't compile (`>= $80`, the `$CE` escapes) to it; `tokenize.py` carries the full R49 escape-statement table, so all X16 leaf statements (VPOKE/SCREEN/LINE/CHAR/COLOR/CLS/FM*/PSG*/SPRITE/…) compile+run via pass-through. **X16 expression FUNCTIONS now work too (done 2026-07-08):** `OP_CALLX`=65 is the expression-context companion to pass-through — GPC evaluates each argument itself (its numeric vars live in a private `varsf` slab, NOT BASIC's table, so a whole `VPEEK(0,X)` can't just go to frmevl), then the runtime formats the computed values as ASCII into a synthesized tokenized call `[$CE][subtok]("("a","b")")`, points TXTPTR at it, and JSRs ROM `frmevl` ($D350); the result comes back in FAC1 and `MOVMF` ($fe66) packs it onto the stack. Escape functions are `$CE $D0..$DE` (the ROM crunch forces a gap after the $80..$C5 statements: `code2.s` `adc #($d0-num_esc_statements-1)`); compiler `is_xfunc` allowlists the numeric-by-value ones (VPEEK $D0, MX/MY/MB $D1-3, JOY $D4, I2CPEEK $D7, MWHEEL $DB) and rejects string-returning HEX$/BIN$/RPT$ + var-ref POINTER/STRPTR + bannex TDATA/TATTR/MOD as a clean SYNTAX error. **Pass-through statements now see GPC variables too (done 2026-07-08):** `parse_passthru` replaces each scalar numeric variable in a `$CE`-escape statement with a `$01<slot>` marker; at run time `op_passthru` splices in the variable's current value as ASCII decimal before handing the statement to `gone3`, so `VPOKE 0,A,V` reaches the ROM as `VPOKE 0,4660,66` and the ROM still does any arithmetic (`VPOKE 0,B*40,V` works). NOT substituted (copied verbatim, unsupported but no corruption): quoted strings, `$`hex/`%`bin literals, string/integer vars (`A$`/`A%`), array/function args (`A(I)`). **Corpus 208/208 PASS.** GOTCHA found + fixed: Prog8 cx16 char literals are PETSCII, so `'a'..'z'` = **$C1..$DA** (shifted letters); `is_alpha()` therefore returns true for token bytes like `$CE` (VPOKE), which made the scanner intern the VPOKE opcode itself as a bogus variable and crash the VM — fixed by an explicit `c >= $80 -> copy verbatim` guard (tokens are never variables). Also raised `PCODE_BASE` `$4800`->`$4C00` (the standalone runtime's buffers now end ~$4852). **String-returning X16 functions HEX$/BIN$/RPT$ done too (2026-07-08):** `OP_CALLXS`=66 — same synthesized-call setup as OP_CALLX, but frmevl returns a STRING; the runtime calls ROM `frestr` ($DE0E) to fetch the result descriptor (len in A, heap body ptr in X/Y) and free the BASIC temp, copies the body off-heap into xbuf, and adopts it into a GPC string temp via `bstr.mem_to_temp` (the same path STR$/CHR$ use). Compiler: `T_XSFUNC` token, `is_xsfunc` allowlists HEX$ $D5 / BIN$ $D6 / RPT$ $DA, added to `is_str_start`, `emit_string_factor` case modeled on T_NSFUNC. Verified incl. two-temp concat + comparison + standalone. **RPT$($DA) done:** `RPT$(<byte>,<count>)` repeats a byte value <count> times — it takes TWO NUMERIC args, NOT a string (my earlier "RPT$ needs string-arg embedding" note misread the ROM `rptd` handler in `x16additions.s`; there is no string argument). So no new machinery was needed: the `emit_string_factor` T_XSFUNC case now parses a counted comma-separated arg list (mirroring the numeric T_XFUNC loop) with an arity check (HEX$/BIN$ = 1 arg, RPT$ = 2), and `xbuild` already formats N comma-separated numeric args. One real fix: `op_callxs` copies the string result into `xbuf`, and RPT$ can return up to 255 bytes, so `xbuf` was enlarged 128->256 (a latent overflow that HEX$/BIN$'s <=16-char results never triggered; runtime BSS still ends $4944, under PCODE_BASE $4C00). Refactored the shared arg-formatting into `vm.xbuild()`. **Corpus 225/225.** **Phase 5 integer-first arithmetic — increment 1 DONE (2026-07-08), the biggest perf lever, ~2.9x on an arithmetic-heavy loop.** Blitz-style COMPILE-TIME integer typing (user chose this over runtime tagging): the compiler tracks each subexpression's type on a type stack that mirrors the runtime stack, and emits native 16-bit integer opcodes for integer-typed work. New `%` integer variables (`A%`, own namespace + `vm.ivarsf` slab; lexer `%` suffix -> `T_IVAR`; `intern_ivar`). Integer opcodes 67-76: IPUSHI/ILOADV/ISTORV/IADD/ISUB/IMUL/INEG + coercions ITOF (top)/ITOF2 (2nd)/FTOI. Runtime `word[32] istack` SHARES `sp` with `float[32] stack` (a slot is int OR float, fixed by the opcode stream -> no runtime tag). Type rules: integer `+ - *` and unary `-` fire only when a real INT (a `%` var) is involved; two integer LITERALS coerce to float, so any `%`-free program emits identical float code -> **corpus green by construction**. `/`, comparisons, FOR, arrays stay float this increment (coerce). Integer math WRAPS at 16 bits (the `%` opt-in, documented divergence); FTOI truncates toward zero (CBM `A%` semantics, `A%=7/2`->3). `emit_op` is the single choke point that picks int-vs-float opcodes + inserts coercions; `parse_expr` auto-coerces its result to float at exit unless `expr_keep_int` (int-var assignment). **Corpus 234/234** (9 `%` tests). GOTCHA fixed (was a real regression): the self-hosted compiler is RAM-tight (MEMTOP $9f00) and the code growth pushed `progend` near MEMTOP, and in-process the string heap was placed at `max(progend,datatop)` = progend -> almost no room -> big strings crashed. FIX in `bstr.init`/`vm`: place the heap at `datatop` (low; the compiler's name-table slabs above it are dead during a run) and CAP the heap ceiling just below `varsf` (lowest LIVE runtime slab) by LOWERING **KERNAL MEMTOP** ($ff99 set) so the ROM `garba2` collector (which re-reads MEMTOP) honours the cap too; standalone `varsf < datatop` so the cap is ignored (full heap). This made the in-process heap MORE robust than before (~2.7 KB dead-slab window vs the tiny progend..MEMTOP one). Also trimmed LIT_SIZE/DATA_SIZE 1024->768 and MAXIVARS=48 to fit the compiler under MEMTOP (banked-RAM name tables are the future headroom lever). PCODE_BASE $4C00->$5000 (runtime slabs grew). **Nested `IF a THEN IF b THEN body` (2026-07-09, branch `harden-and-tier2`, compiler-only).** Previously `parse_then_body`
rejected a leading IF because `parse_if` couldn't re-enter itself (Prog8 has no recursion). A chain of leading IFs is a
conjunction of guards that all skip to the SAME point (just past the one shared body), so any false condition skips it.
`parse_if` now LOOPS over each `IF <cond> THEN`, emitting a JZ/IJZ per guard + collecting its patch slot (`ifpatch[MAXIFNEST=8]`),
until the token after THEN isn't another IF; then parses one then-body and backpatches every guard to the body end. Each guard
independently picks IJZ vs JZ by its own condition type (int/float mix freely). No new opcodes, no runtime change (so it did NOT
grow the runtime footprint). +9 corpus tests (both-true/outer-false/inner-false, triple nest, mixed int+float guards, GOTO-shorthand
innermost body, nested-IF-driven loop, standalone). This session also shipped the build-time PCODE_BASE assert + a PCODE_BASE
$3e00->$3c80 tightening — see [[gpc-runtime-asm-conversion]].
